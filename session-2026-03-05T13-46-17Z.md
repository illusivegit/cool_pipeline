## Strategy Plan

### Principles

**1. Scrape what you operate, not what you can reach.**
Every scrape target should answer the question: "If this metric changes, would I take action?" If the answer is no, don't scrape it. For your stack today, Grafana's internal metrics (like dashboard render times) are *nice to have* but unlikely to page anyone. Keep them for now since the cardinality is low, but don't build alerts on them.

**2. Golden signals per layer, not per container.**
The four golden signals (latency, traffic, errors, saturation) apply differently at each layer:

| Layer | Latency | Traffic | Errors | Saturation |
|-------|---------|---------|--------|------------|
| **Edge (Nginx)** | Upstream response time | Requests/sec | 4xx/5xx rate | Connections (active/waiting) |
| **Service (Flask)** | `http_request_duration_seconds` | `http_requests_total` rate | `http_errors_total` rate | In-flight requests |
| **Data (SQLite)** | `db_query_duration_seconds` | Queries/sec (from histogram count) | N/A (errors bubble up to HTTP 5xx) | N/A for SQLite |
| **Host (VM)** | N/A | Network bytes/sec | N/A | CPU%, memory%, disk% |
| **Observability Stack** | Query latency (Loki, Tempo) | Ingestion rate | Dropped spans/logs | Memory/disk usage |

**3. SLO-first alerting.**
Define the SLO, then alert on the burn rate. "P95 latency < 500ms" is a better starting point than "alert if any request is slow." You already have the P95 alert -- good. Next step is burn-rate alerts for availability SLOs.

**4. Cardinality budget.**
For a small lab stack, aim for < 50,000 active time series. Each additional label dimension multiplies cardinality. The `path` label in promtail is the biggest risk today -- replace it with `path_group` (see Phase 0).

**5. Ownership = job label.**
The `job` label is the natural ownership boundary. Don't add redundant `service` labels that duplicate `job`. When you need cross-job aggregation, use `group_left`/`group_right`, not denormalized labels.

### Phased Roadmap

#### Phase 0: Harden What You Have (effort: ~1 hour)

**What to do:**
1. **Remove redundant `service` labels** from `prometheus.yml` scrape configs. The `job` label already identifies each target. Keeping both wastes label storage and confuses queries.
2. **Fix promtail `path` label cardinality**. Replace the raw `path` extraction with a `replace` stage that normalizes paths to groups (e.g., `/api/tasks/123` -> `/api/tasks/:id`). Or drop `path` from labels entirely and keep it only in the log line.
3. **Add `honor_labels: true`** to the `flask-backend` scrape job. Your app sets its own labels via `prometheus_client`; Prometheus should respect them.
4. **Add `scrape_timeout: 10s`** explicitly to match the default (defensive clarity).

**What NOT to change:**
- Don't change scrape intervals. 15s is the right default for this stack size.
- Don't remove Grafana or Alertmanager scrape jobs -- low cardinality, useful for debugging.

#### Phase 1: Add Edge Metrics via Nginx stub_status (effort: ~30 minutes)

**Recommendation: Use `stub_status` + community nginx-prometheus-exporter sidecar.**

Why not the Nginx VTS module or OpenResty? Because your frontend is a stock `nginx:alpine` image. `stub_status` is built-in, zero-dependency, and gives you the four most important edge metrics:
- Active connections
- Accepts/handled/requests (cumulative counters)
- Reading/writing/waiting connection states

For richer HTTP metrics (request rate by status code, upstream latency), you'd need a custom Nginx build or the OTEL Nginx module. But **you already get HTTP status codes from promtail log parsing** -- so `stub_status` fills the gap (connection-level metrics) without duplicating what promtail gives you.

**Implementation:**
1. Add a `stub_status` location to `default.conf` (internal only, not exposed to users).
2. Add `nginx/nginx-prometheus-exporter` as a sidecar service in `docker-compose.yml`.
3. Add a scrape job in `prometheus.yml`.

#### Phase 2: Add Container Metrics via cAdvisor (effort: ~30 minutes)

**Recommendation: Yes, add cAdvisor.**

Why cAdvisor over alternatives?
- **Docker stats API**: Only gives point-in-time snapshots; no Prometheus exposition.
- **Prometheus Docker SD**: Discovers targets, doesn't collect container resource metrics.
- **cAdvisor**: Purpose-built for container resource metrics, Prometheus-native, widely used.

What it gives you that node-exporter does NOT:
- Per-container CPU usage, memory usage, network I/O, disk I/O
- Container restart counts
- Container resource limit utilization (actual vs. limit from `deploy.resources`)

This is critical for answering: "Is the OTel Collector hitting its 512M memory limit?" or "Is Prometheus CPU-throttled?"

**Cardinality note:** cAdvisor can be chatty. Use `metric_relabel_configs` to keep only the metrics you need (about 10-15 metric names out of 100+).

#### Phase 3: Alerting Improvements (effort: ~1-2 hours)

**Current state:** 7 alert rules. Good coverage of basics. What's missing:

1. **Availability burn-rate alert** (SLO-based). More useful than a static error rate threshold because it adapts to traffic volume.
2. **Loki/Tempo ingestion health**. You alert on OTel dropped spans, but not on Loki push failures or Tempo ingestion failures.
3. **Prometheus self-monitoring**. TSDB health, rule evaluation failures, scrape duration.
4. **Disk prediction**. Linear prediction of disk exhaustion (more useful than a static 80% threshold).

**What to avoid:**
- Don't alert on every `up == 0`. The existing `PrometheusTargetDown` already covers all targets. Don't duplicate with per-job `up` alerts (that's what the BackendDown rule does -- it's redundant with PrometheusTargetDown).
- Don't set `for: 0s`. Always require at least 1 minute of sustained failure to avoid transient blips.
- Don't alert on Grafana being down. Grafana is a read path; if it's down, nobody can see alerts in Grafana anyway. You'd need an external notification (Alertmanager webhook) to be useful.

## Alerts

### Recommended Alert Rules (High-Signal, Low-Noise)

Below are 10 alert rules organized by priority. Rules 1-7 are the core set; 8-10 are stretch goals.

#### Rule 1: Availability SLO Burn Rate (5m window)

**Rationale:** Better than a static error rate threshold. A 2% error rate on 10 req/s is very different from 2% on 10,000 req/s. Burn-rate alerts fire when you're consuming your error budget faster than expected.

```yaml
- alert: HighAvailabilityBurnRate
  expr: |
    (
      sum(rate(http_errors_total[5m]))
      /
      clamp_min(sum(rate(http_requests_total{endpoint!="metrics"}[5m])), 1e-10)
    ) > 14.4 * (1 - 0.99)
  for: 2m
  labels:
    severity: critical
  annotations:
    summary: "Availability SLO burn rate is critical"
    description: "Error rate is burning through the 99% availability SLO at 14.4x the sustainable rate over the last 5 minutes."
```

The `14.4` multiplier comes from Google SRE: if your SLO is 99% (error budget = 1%), alerting at 14.4x burn rate over 5 minutes means you'd exhaust your monthly error budget in ~1 hour. This catches severe incidents fast.

#### Rule 2: Backend Down (KEEP EXISTING, but improve)

```yaml
- alert: BackendDown
  expr: up{job="flask-backend"} == 0
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "Flask backend is unreachable"
    description: "Prometheus cannot scrape the flask-backend /metrics endpoint. The application may be crashed or the container may be unhealthy."
```

**Keep this alongside PrometheusTargetDown** because `BackendDown` fires after 1 minute (fast) with `severity: critical`, while `PrometheusTargetDown` fires after 2 minutes with `severity: warning`. The inhibit rule in alertmanager.yml will suppress the warning when the critical fires.

#### Rule 5: Prometheus TSDB Approaching Disk Limit

```yaml
- alert: PrometheusTSDBHighDiskUsage
  expr: |
    (prometheus_tsdb_storage_blocks_bytes + prometheus_tsdb_wal_storage_size_bytes)
    / (1024 * 1024 * 1024) > 4
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "Prometheus TSDB using more than 4GB disk"
    description: "Prometheus storage is at {{ $value | humanize }}GB. Review retention settings or cardinality."
```

**Rationale:** On a small VM, TSDB growth can fill disk silently. 4GB is a reasonable threshold for a lab with 15d retention.

#### Rule 6: Host Disk Exhaustion Prediction

```yaml
- alert: HostDiskWillFillIn24h
  expr: |
    predict_linear(node_filesystem_avail_bytes{mountpoint="/",fstype=~"ext4|xfs|btrfs"}[6h], 24*3600) < 0
  for: 30m
  labels:
    severity: warning
  annotations:
    summary: "Host disk predicted to fill within 24 hours"
    description: "Based on the last 6 hours of disk usage trend, the root filesystem will be full in less than 24 hours."
```

**Rationale:** Much more useful than a static 80% threshold. A disk at 85% that's stable is fine; a disk at 60% that's growing linearly is not.

### Anti-Noise Tactics

1. **Always use `for:` durations.** Minimum 1m for critical, 2-5m for warning. Never 0.
2. **Don't alert on `up == 0` per job.** The generic `PrometheusTargetDown` covers everything. Only add per-job `up` alerts for critical-path services (like BackendDown) where you want faster/different routing.
3. **Don't alert on Grafana health.** If Grafana is down, you can't see Grafana alerts. Use Alertmanager webhook to an external system.
4. **Don't alert on high cardinality metrics.** If a metric has >1000 time series, aggregating it in an alert rule is expensive and noisy.
5. **Use inhibit rules.** Your existing rule (critical suppresses warning on same alertname) is correct. Consider adding: if the host is down (HostDown), suppress all container/app alerts for that host.
6. **Group by service, not by instance.** In `alertmanager.yml`, `group_by: ['alertname', 'severity']` is correct for your single-instance lab. Don't add `instance` -- it would prevent grouping.

### What to Remove or Consolidate

Consider that `BackendDown` and `PrometheusTargetDown` overlap. Options:
- **Keep both** (current): BackendDown fires faster (1m) with critical severity. PrometheusTargetDown fires at 2m with warning. The inhibit rule prevents double-notification. This is fine.
- **Remove BackendDown** and change PrometheusTargetDown to route critical for `job="flask-backend"`: cleaner, but requires alertmanager routing changes.

Recommendation: keep both for now. It's clear and explicit.

## Explicit "Do NOT Do This" List

1. **Do NOT scrape everything just because you can.** Every scrape target adds cardinality, memory usage, and disk cost. If you can't name a specific alert or dashboard panel that would use a metric, don't scrape it.

2. **Do NOT alert on every `up == 0`.** The blanket `PrometheusTargetDown` rule already catches all targets. Adding per-job `up` alerts for non-critical services (Grafana, promtail) creates noise.

3. **Do NOT use high-cardinality labels in promtail.** The `path` label with raw URL paths (e.g., `/api/tasks/1`, `/api/tasks/2`, ...) creates a new time series for every unique path. This is the #1 cardinality killer in Loki setups. Normalize or drop it.

4. **Do NOT add `instance` to alertmanager `group_by`.** In a single-instance lab, this would prevent alert grouping and spam you with duplicate notifications.

5. **Do NOT use `for: 0s` on alerts.** A single failed scrape or a momentary spike will trigger the alert. Always require sustained failure (1-5 minutes minimum).

6. **Do NOT expose `stub_status` to the public.** Restrict it to Docker-internal networks. It leaks server internals (connection counts, total requests).

7. **Do NOT use the OTel Collector to export Prometheus metrics AND scrape Flask directly.** You correctly chose one path (direct scrape). Keeping both would double your metric volume.

8. **Do NOT create a dashboard for every component.** Grafana dashboards are cheap to create and expensive to maintain. Every dashboard needs someone to look at it. If nobody looks at a "Promtail Health" dashboard, don't build it. Build the Overview dashboard first and add detail dashboards only when the overview isn't enough.

9. **Do NOT set scrape_interval below 15s** for a lab stack. Faster scraping increases load on both Prometheus and the targets with negligible improvement in signal quality. The only exception would be if you need sub-15s resolution for SLO calculations (you don't, at lab scale).

10. **Do NOT add a database exporter for SQLite.** SQLite doesn't have a network protocol to export metrics from. Your app already exposes `db_query_duration_seconds` via prometheus_client -- that's the right approach. A database exporter is for PostgreSQL/MySQL/etc. where the DB is a separate service.

### Issues Encountered and Fixed During Testing

1. **Docker Desktop bind mount cache**: After editing `prometheus.yml` on the host, `docker restart prometheus` was not picking up changes because Docker Desktop caches bind mounts at container creation. Fix: `docker compose restart prometheus` forces a fresh mount sync.

2. **cAdvisor on Docker Desktop**: cAdvisor cannot see individual container cgroups on Docker Desktop (Linux) because Docker Desktop runs containers inside a LinuxKit VM. The `/var/lib/docker/image/overlayfs/layerdb/mounts/` paths don't exist from the host perspective. cAdvisor only exposes cgroup-level aggregates (`/docker` = all containers combined, `/docker/buildkit`, etc.). Fix: Updated `metric_relabel_configs` to filter on `id` label instead of `name`, and updated dashboard queries to use `id=~"/docker.*"`. On a native Docker host (or real VM deployment), per-container metrics with `name` labels would be available.

3. **cAdvisor `honor_timestamps: false`**: cAdvisor metrics include millisecond timestamps. Added `honor_timestamps: false` to the scrape config to let Prometheus assign its own scrape timestamps, avoiding potential timestamp ordering issues.

4. **Containers dashboard JSON wrapper**: The agent that built `containers.json` wrapped the dashboard in a `{"dashboard": {...}, "overwrite": true}` structure (API-style payload). Grafana's file provisioner expects the dashboard object at the top level. Fix: Unwrapped the JSON.

### Fix 1: Loki panels "No data" in Service Health (HTTP Status Codes, Top Paths, Nginx Error Logs)

**Symptom**: Three Loki-sourced panels in Service Health dashboard showed "No data".

**Root cause**: Promtail's pipeline used `source: filename` in a `regex` stage to extract `log_type` from the filename. However, `filename` is a Promtail *label*, not *extracted data* -- the `source` field in a regex stage only reads from the extracted data map. The regex silently matched nothing, so `log_type` was never set. Loki received logs without `log_type`, and the dashboard queries `{job="nginx", log_type="access"}` returned empty.

**Fix** (`otel-collector/promtail-config.yml`): Replaced the single scrape job with two separate jobs that set `log_type` as a static label:
- `nginx-access`: `__path__: /var/log/nginx/access.log`, `log_type: access`
- `nginx-error`: `__path__: /var/log/nginx/error.log`, `log_type: error`

Pipeline stages for access log parsing (regex, labels, replace for path normalization) and error log parsing (regex for level) remain on their respective jobs.

**Verified**: `{job="nginx", log_type="access"}` returns streams with `status`, `method`, `path_group` labels correctly extracted.

### Fix 2: Spans Exported/sec and Failed Spans/sec "No data" in Infrastructure

**Symptom**: Two panels in the Observability Stack row showed "No data".

**Root cause**: Dashboard queries used `otelcol_exporter_sent_spans_total` and `otelcol_exporter_send_failed_spans_total`, but the OTel Collector exposes these metrics without the `_total` suffix: `otelcol_exporter_sent_spans` and `otelcol_exporter_send_failed_spans`.

**Fix**: Removed `_total` suffix in three files:
- `grafana/dashboards/infrastructure.json` -- both span panels
- `grafana/dashboards/overview.json` -- OTel Dropped Spans/min panel
- `otel-collector/alert-rules.yml` -- `OtelCollectorDroppedSpans` alert rule

**Verified**: Both panels now render span rate data.

### Fix 3: Container Network RX/TX "No data" in Infrastructure

**Symptom**: Two container network panels showed "No data".

**Root cause**: Docker Desktop does not expose per-container network metrics via cAdvisor. Network metrics only exist at the root cgroup (`id="/"`), but the Prometheus `metric_relabel_configs` for the cadvisor job dropped `id="/"`:
```yaml
regex: /|/kubepods.*|/podruntime.*|/procd-paused|/restricted
action: drop
```
Node-exporter also couldn't help because it lacks `network_mode: host`, so it only sees its own container's `eth0`.

**Fix** (two changes):
1. `otel-collector/prometheus.yml`: Removed bare `/` from the cadvisor drop regex, allowing root cgroup metrics through. CPU/memory panels are unaffected because they filter by `id=~"/docker.*"`.
   ```yaml
   regex: /kubepods.*|/podruntime.*|/procd-paused|/restricted
   ```
2. `grafana/dashboards/infrastructure.json`: Changed panel queries from `container_network_{receive,transmit}_bytes_total{id=~"/docker.*"}` to:
   ```
   rate(container_network_receive_bytes_total{id="/", interface=~"eth0|br-.*"}[5m])
   rate(container_network_transmit_bytes_total{id="/", interface=~"eth0|br-.*"}[5m])
   ```
   Panels renamed to "Container Network RX/TX" with description noting the Docker Desktop limitation. Legend shows per-interface breakdown (`eth0`, `br-*` bridges).

**Verified**: 5 series per panel (4 Docker bridge interfaces + eth0) with live traffic data.

### Note: Container Network RX/TX Legend Explained

### Question: what are "br-02b719cd0d37, br-1656f5bb18a5, br-b76d61cfa842, br-f01fc63eb352, eth0" in Container Network RX/TX and what does RX/TX mean?

- **RX/TX** = Receive/Transmit. Inbound vs outbound network traffic in bytes per second.
- **`eth0`** = the host's primary network interface (internet-facing traffic).
- **`br-*`** = Linux bridge interfaces, one per Docker network. Each `br-<hash>` corresponds to a network defined in `docker-compose.yml`. The 3 compose networks (`frontend-net`, `backend-net`, `observability`) plus the default bridge account for the 4 `br-*` interfaces. The hex suffix is Docker's internal ID -- not human-readable, but each maps 1:1 to a compose network.
- Docker Desktop doesn't expose per-container network metrics -- only the root cgroup aggregate. So instead of "grafana sent X bytes", you see "the observability bridge network carried X bytes."

## Dashboard Usage Guide

### Overview -- System Health
**Purpose**: Single-pane-of-glass. Answers "is anything broken right now?" in 5 seconds.

**Use it**: As your landing page. Glance at the four top stats (availability, P95 latency, targets up, host CPU). If anything is yellow/red, drill into the relevant dashboard below. The bottom row shows observability stack health -- dropped spans, head series, active alerts, disk -- so you know if your monitoring itself is healthy.

### Service Health
**Purpose**: Application-layer deep dive across both the backend (Flask) and the edge (Nginx).

**Use it when**: The Overview shows degraded availability or high latency. The **Backend SLIs** row tells you *what's broken* -- which endpoints have errors, what latency percentiles look like, which status codes are spiking. The **Edge / Nginx** row tells you *what users see* -- connection states, request rate, HTTP status codes and top paths from actual access logs, plus error logs at the bottom for immediate context.

**Workflow**: Overview shows error rate spike -> open Service Health -> Backend SLIs row shows `/api/tasks` endpoint at 15% errors -> scroll to Edge/Nginx -> error logs show upstream timeouts -> you know the backend is slow, not the proxy.

### Infrastructure
**Purpose**: Resource-layer deep dive across host, containers, and the observability stack itself.

**Use it when**: You suspect resource pressure (OOM, CPU throttling, disk filling) or need to verify the monitoring pipeline is healthy. Three rows:
- **Host** -- CPU, memory, disk, network, load averages. Is the VM itself under pressure?
- **Containers** -- Docker cgroup CPU/memory, memory vs limits, network per bridge interface. Which containers are hungry?
- **Observability Stack** -- TSDB head series (cardinality), scrape durations, rule eval time, span export rates, Loki/Tempo ingestion, collector memory RSS with threshold lines, and a scrape targets UP/DOWN table.

**Workflow**: Overview shows host CPU at 85% -> open Infrastructure -> Host row confirms sustained load -> Containers row shows `/docker` cgroup at 70% CPU -> Obs Stack row shows TSDB head series jumped to 80k -> you know a cardinality explosion is causing Prometheus to consume resources.

### End-to-End Tracing
**Purpose**: Request-level debugging. Follow a single request through the entire system.

**Use it when**: You've identified a problematic endpoint in Service Health and need to understand *why* it's slow or failing. The trace search table lets you filter by service, duration, or status. Click a trace ID to see the full span waterfall in Tempo. The service dependency map shows how services connect. Application logs at the bottom are correlated by trace ID.

**Workflow**: Service Health shows P99 latency spike on `/api/tasks` -> open End-to-End Tracing -> search for slow traces (`{ resource.service.name = "flask-backend" && duration > 500ms }`) -> click a trace -> waterfall shows the DB query span took 450ms -> root cause identified.

### Drill-Down Summary

Overview first, always. Then drill down: Service Health for app problems, Infrastructure for resource problems, End-to-End Tracing for request-level debugging.

## Note: Browser Traces Go Directly to the OTel Collector

### Question: Does the browser send OTLP traces through Nginx and the backend, or directly to the Collector?

The browser's OTel JS SDK sends traces **directly** to the OTel Collector at `:4318/v1/traces` -- it does NOT go through Nginx or the backend. A single user action (e.g. "create task") produces two parallel HTTP requests:

```
Browser --(POST /api/tasks)---> Nginx ---> Flask Backend
Browser --(POST /v1/traces)---> OTel Collector (port 4318, CORS enabled)
```

The Collector has CORS configured to accept these cross-origin requests. The browser trace and backend trace are correlated via a shared trace ID propagated through the `traceparent` HTTP header.

### Security Concern: Does exposing the Collector to the browser open it to exploitation?

Yes, this is a real attack surface. Port 4318 is publicly reachable and accepts arbitrary OTLP payloads from any browser or client. Risks include:

- **Trace/log injection**: An attacker can send crafted spans with fake service names, misleading attributes, or enormous payloads to pollute Tempo and downstream dashboards.
- **Resource exhaustion**: Flooding the Collector's OTLP/HTTP endpoint with high-volume garbage spans could overwhelm the batch processor, exhaust its 512 MiB memory limiter, and cascade into Tempo storage pressure.
- **No authentication**: The Collector's OTLP receiver has no auth -- any HTTP client can POST to it. In production, this is mitigated by:
  - Putting the Collector behind Nginx with rate limiting on the `/v1/traces` path
  - Adding an API key or token header check (OTel Collector supports `headerssetter` and auth extensions)
  - Restricting CORS `allowed_origins` to only the application domain (currently accepts all origins)
  - Network-level controls (WAF, IP allowlisting) if the Collector isn't meant to be internet-facing

For this lab stack, the risk is acceptable. For production, the Collector's public OTLP endpoint should be hardened.

### Verified: Current CORS Configuration is Wide Open

Inspected `otel-collector/otel-collector-config.yml` -- the OTLP/HTTP receiver CORS config:

```yaml
cors:
  allowed_origins:
    - "http://localhost:8080"
    - "http://*"
  allowed_headers:
    - "*"
```

The wildcard `"http://*"` means **any HTTP origin** can send OTLP data to the Collector. Combined with `allowed_headers: ["*"]`, there are zero browser-enforced restrictions. Any webpage on the internet could POST spans to this endpoint if port 4318 is reachable.

**Production hardening checklist:**

1. **Lock down CORS `allowed_origins`** -- replace `"http://*"` with the exact application domain(s) only (e.g., `"https://app.example.com"`).
2. **Rate limit the `/v1/traces` path** -- put the Collector behind Nginx/a reverse proxy with `limit_req` to cap requests per second per IP.
3. **Add authentication** -- use the OTel Collector's `bearertokenauth` or `headerauthextension` to require an API key on every OTLP request. The browser SDK can inject the key via `headers` config.
4. **Network controls** -- if the Collector should only accept browser traces from known users, use a WAF or IP allowlist. If it should only accept internal telemetry, don't expose port 4318 publicly at all.
5. **Payload size limits** -- configure `max_recv_msg_size_mib` on the OTLP receiver to reject oversized payloads before they hit the batch processor.

## Phase 7 -- OTel Collector Hardening (CORS + Rate Limiting)

### Problem

Port 4318 (OTLP/HTTP) was publicly exposed and accepted arbitrary payloads from any origin. The browser SDK sent traces directly to `http://<hostname>:4318/v1/traces`, bypassing Nginx entirely. CORS was configured with a wildcard `"http://*"` that allowed any webpage to POST spans.

### Solution: Two changes implemented

#### 1. CORS origin from `.env` (Option C)

Instead of hardcoding origins or using a wildcard, the allowed origin is pulled from `.env` via OTel Collector environment variable substitution.

**`.env`** -- added:
```
ALLOWED_ORIGIN=http://localhost:8080
```

**`otel-collector/otel-collector-config.yml`** -- replaced wildcard:
```yaml
# Before
cors:
  allowed_origins:
    - "http://localhost:8080"
    - "http://*"

# After
cors:
  allowed_origins:
    - "${env:ALLOWED_ORIGIN}"
```

**`docker-compose.yml`** -- passed env var to collector container:
```yaml
environment:
  - ALLOWED_ORIGIN=${ALLOWED_ORIGIN:-http://localhost:8080}
```

For VM deployments where the host accesses the app via the VM's IP, set `ALLOWED_ORIGIN=http://<VM_IP>:8080` in `.env`. The browser's `Origin` header is the URL in the address bar, not `localhost`.

#### 2. Rate-limited trace proxy through Nginx

Rerouted browser traces through Nginx with rate limiting, then removed the public port 4318 mapping entirely.

**`frontend/default.conf`** -- added rate limit zone and proxy location:
```nginx
# At http context (top of file, outside server block)
limit_req_zone $binary_remote_addr zone=traces:1m rate=10r/s;

# Inside server block
location /v1/traces {
    limit_req zone=traces burst=20 nodelay;
    limit_req_status 429;

    set $collector_upstream http://otel-collector:4318;
    proxy_pass $collector_upstream;

    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header Content-Type $content_type;

    proxy_connect_timeout 5s;
    proxy_read_timeout 10s;
}
```

Rate limit: 10 requests/second per IP with burst of 20. Exceeding clients receive HTTP 429.

**`frontend/otel-browser.js`** -- changed to same-origin relative URL:
```js
// Before
const collectorUrl = `http://${window.location.hostname}:4318/v1/traces`;

// After
const collectorUrl = '/v1/traces';
```

Same-origin request eliminates CORS entirely for the browser→Nginx path. Nginx→Collector is server-to-server (no CORS).

**`docker-compose.yml`** -- removed public port 4318:
```yaml
# Before
ports:
  - "${OTEL_GRPC_PORT:-4317}:4317"
  - "${OTEL_HTTP_PORT:-4318}:4318"
  - "8888:8888"

# After
ports:
  - "${OTEL_GRPC_PORT:-4317}:4317"
  - "8888:8888"
```

Port 4318 remains accessible within Docker networks (backend uses `http://otel-collector:4318` internally). Only external/browser access is removed.

### New trace flow

```
Before:  Browser --(POST :4318/v1/traces)--> OTel Collector (direct, no auth, no rate limit)
After:   Browser --(POST :8080/v1/traces)--> Nginx (rate-limited) --> OTel Collector
```

### Network connectivity

Nginx (`frontend`) reaches `otel-collector` via the shared `backend-net` Docker network. No network changes were needed.
