# Jenkins Plugin Requirements for Jenkinsfile

## Setup Context

- **Controller**: `jenkins/jenkins:lts-jdk21` — vanilla, no pre-installed plugins beyond defaults
- **Agent**: Custom inbound agent (`jenkins/inbound-agent:latest` + jq)
- **Agent label**: `docker-agent1` (inbound/JNLP — no extra plugin needed, protocol is built into Jenkins core)

## Required Plugins

| Plugin | ID | Why |
|---|---|---|
| **Pipeline** | `workflow-aggregator` | Umbrella plugin that enables the entire `pipeline {}` DSL. Pulls in sub-plugins for `sh`, `echo`, `archiveArtifacts`, stage steps, etc. |
| **Timestamper** | `timestamper` | `options { timestamps() }` on line 3 |
| **SSH Agent** | `ssh-agent` | `sshagent(credentials: ['vm-ssh'])` used in 7 stages/post blocks (lines 27, 41, 53, 67, 82, 110, 123, 139, 151) |
| **Credentials** | `credentials` | Storing the `vm-ssh` SSH private key. Dependency of SSH Agent but must be present. |

### Bundled by `workflow-aggregator` (no separate install needed)

These sub-plugins are automatically installed with Pipeline but are listed here so you know what provides each DSL step:

| Sub-plugin | ID | Provides |
|---|---|---|
| Pipeline: Basic Steps | `workflow-basic-steps` | `archiveArtifacts`, `echo`, `error` |
| Pipeline: Nodes and Processes | `workflow-durable-task-step` | `sh` step |
| Pipeline: Groovy CPS | `workflow-cps` | Groovy CPS execution engine |
| Pipeline: Job | `workflow-job` | Pipeline job type |
| Pipeline: Stage Step | `pipeline-stage-step` | `stage()` blocks |
| Pipeline: SCM Step | `workflow-scm-step` | SCM checkout in pipelines |

## Optional (recommended)

| Plugin | ID | Why |
|---|---|---|
| Pipeline: Stage View | `pipeline-stage-view` | Visual stage-by-stage progress in the Jenkins UI |
| Git | `git` | If the pipeline job is configured to pull from a Git SCM source |

## Minimum Install Command

Using the Jenkins CLI (from inside the controller container):

```bash
jenkins-plugin-cli --plugins \
  workflow-aggregator \
  timestamper \
  ssh-agent \
  credentials
```

Or if you prefer to bake them into a custom controller image:

```dockerfile
FROM jenkins/jenkins:lts-jdk21
RUN jenkins-plugin-cli --plugins \
  workflow-aggregator \
  timestamper \
  ssh-agent \
  credentials \
  pipeline-stage-view \
  git
```

## Credential Setup

After plugins are installed, create the `vm-ssh` credential:

1. **Manage Jenkins > Credentials > System > Global credentials**
2. **Kind**: SSH Username with private key
3. **ID**: `vm-ssh`
4. **Username**: `jenkins` (matches `VM_USER` in the Jenkinsfile)
5. **Private Key**: Enter the private key that has access to the VM at `192.168.122.230`

# Alertmanager Email Alerting — Architecture Decision & Implementation

## Problem

The Docker-CICD Alertmanager had stub receivers (`webhook_configs: []`) — alerts fired but went nowhere. The goal: wire up real email delivery so alerts reach a mailbox.

## Options Evaluated

### Option A: Alertmanager → Yahoo SMTP directly (CHOSEN)

Alertmanager has native SMTP+TLS+SASL support. Configure `smtp_smarthost`, `smtp_auth_username`, `smtp_auth_password` directly in alertmanager.yml and let it talk to Yahoo's SMTP relay without an intermediary.

**Pros:**
- Zero additional services — no new container, no new image, no new attack surface
- Alertmanager handles TLS negotiation and SASL auth natively
- No sender-rewrite needed — `smtp_from` is set explicitly in the config, so Yahoo always sees the correct `MAIL FROM`
- Matches 12-factor: secrets injected at build time via `.env.secrets`

**Cons:**
- SMTP credentials live in the rendered config file (mitigated: rendered file is gitignored)
- If you switch SMTP providers, you edit the template rather than swapping a relay container

### Option B: Add a Postfix container to the Compose stack

Add a `postfix` service to docker-compose on the `observability` network. Alertmanager sends to `postfix:25`; Postfix relays upstream with TLS+SASL.

**Pros:**
- Decouples SMTP provider credentials from Alertmanager config
- Matches the Infrastructure Automation Framework's VM pattern (Alertmanager → localhost:25 → Postfix → Yahoo)
- Single place to change relay config if multiple services need to send email

**Cons:**
- Adds a 13th container for a single-purpose relay
- Requires a custom Dockerfile for Postfix (not available as a simple pull-and-run image)
- Introduces sender-rewrite complexity (the exact problem that burned 45 minutes on the host — `sender_canonical_maps`, `smtp_header_checks`)
- More moving parts to debug when email delivery fails
- Overkill: only Alertmanager sends email in this stack

### Why Option A wins for Docker-CICD

The Infrastructure Automation Framework uses Postfix because Alertmanager runs as a systemd service on bare metal — separating SMTP relay into its own daemon is good Unix practice. But in Docker Compose, every service is already isolated. Adding a container just to relay SMTP from one container to Yahoo adds complexity with no architectural benefit. Alertmanager can talk TLS+SASL to Yahoo directly, and it controls its own `smtp_from` so there's no sender-rewrite problem.

The host-side Postfix debugging session proved the point: `sender_canonical_maps` with regexp catch-all, `smtp_header_checks`, envelope vs header rewriting — all of that complexity exists because `mail` (or any non-Alertmanager sender) doesn't control its `MAIL FROM`. Alertmanager does. Skip the middleman.

## Implementation

### Files created

| File | Purpose |
|---|---|
| `.env.secrets` | SMTP credentials + alert recipient (gitignored) |
| `.env.secrets.example` | Shows users what vars to populate (tracked) |
| `otel-collector/alertmanager.yml.tmpl` | Email-enabled Alertmanager config template (tracked) |

### Files modified

#### `.env` — added alerting variables

```diff
+# --- Alerting (optional — requires .env.secrets) ----------------------------
+ENABLE_ALERTING=true
+SMTP_RELAYHOST=[smtp.mail.yahoo.com]:587
+SMTP_FROM=walidahmm@yahoo.com
+
 # --- Project -----------------------------------------------------------------
```

Non-secret relay config lives here. Secrets stay in `.env.secrets`.

#### `.env.secrets` — created (gitignored)

```
SMTP_AUTH_USERNAME=walidahmm@yahoo.com
SMTP_AUTH_PASSWORD=<app-password>
ALERT_EMAIL_TO=walidahmm@yahoo.com
```

#### `.gitignore` — added secret/rendered files

```diff
+# Secrets
+.env.secrets
+
+# Rendered alertmanager config (contains secrets from .env.secrets)
+otel-collector/.alertmanager-rendered.yml
+
 # IMPORTANT: Private documentation archive (strategic IP protection)
```

#### `otel-collector/alertmanager.yml.tmpl` — created (tracked)

```yaml
global:
  resolve_timeout: 5m
  smtp_smarthost: '${SMTP_RELAYHOST}'
  smtp_from: '${SMTP_FROM}'
  smtp_auth_username: '${SMTP_AUTH_USERNAME}'
  smtp_auth_password: '${SMTP_AUTH_PASSWORD}'
  smtp_require_tls: true

route:
  group_by: ['alertname', 'severity']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h
  receiver: 'default'
  routes:
    - match:
        severity: critical
      receiver: 'critical'
      group_wait: 5s
    - match:
        severity: warning
      receiver: 'default'

receivers:
  - name: 'default'
    email_configs:
      - to: '${ALERT_EMAIL_TO}'
        send_resolved: true
        headers:
          Subject: '[Lab] {{ .Status | toUpper }}: {{ .GroupLabels.alertname }}'

  - name: 'critical'
    email_configs:
      - to: '${ALERT_EMAIL_TO}'
        send_resolved: true
        headers:
          Subject: '[Lab] CRITICAL: {{ .GroupLabels.alertname }}'

inhibit_rules:
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['alertname']
```

Uses `${VAR}` placeholders rendered by `envsubst` at build time. Alertmanager Go template syntax (`{{ .Status }}`) is unaffected because `envsubst` only substitutes `$VAR`/`${VAR}` patterns.

#### `docker-compose.yml` — changed alertmanager volume mount

```diff
-      - ./otel-collector/alertmanager.yml:/etc/alertmanager/alertmanager.yml:ro
+      - ./otel-collector/.alertmanager-rendered.yml:/etc/alertmanager/alertmanager.yml:ro
```

Mounts the rendered config (which contains real credentials) instead of the stub. The rendered file is gitignored.

#### `Makefile` — added config rendering, wired into `up`

```diff
 # Load .env if present (image tags, ports, retention, project name)
 -include .env
+-include .env.secrets
 export

+# ── Config Rendering ──────────────────────────────────────────────────────
+
+.PHONY: render-alertmanager
+
+render-alertmanager:
+	@if [ "$(ENABLE_ALERTING)" = "true" ] && [ -f .env.secrets ]; then \
+		set -a && . ./.env && . ./.env.secrets && set +a && \
+		envsubst < otel-collector/alertmanager.yml.tmpl > otel-collector/.alertmanager-rendered.yml && \
+		echo "Alertmanager config rendered (alerting enabled)"; \
+	else \
+		cp otel-collector/alertmanager.yml otel-collector/.alertmanager-rendered.yml && \
+		echo "Alertmanager config copied (alerting disabled — no .env.secrets or ENABLE_ALERTING!=true)"; \
+	fi

-up: ## Build and start all services
+up: render-alertmanager ## Build and start all services
```

`make up` now renders the alertmanager config first. If `.env.secrets` is missing or `ENABLE_ALERTING!=true`, it falls back to the stub config (empty webhook receivers, no email).

### Data flow

```
make up
  │
  ├─ render-alertmanager
  │    ├─ .env.secrets exists + ENABLE_ALERTING=true?
  │    │    YES → envsubst: alertmanager.yml.tmpl → .alertmanager-rendered.yml (with secrets)
  │    │    NO  → cp: alertmanager.yml → .alertmanager-rendered.yml (stub, no email)
  │    │
  │    └─ .alertmanager-rendered.yml is gitignored (secrets never committed)
  │
  └─ docker compose up
       └─ alertmanager container mounts .alertmanager-rendered.yml
            └─ Alertmanager → smtp.mail.yahoo.com:587 (TLS + SASL)
                 └─ Email delivered to ALERT_EMAIL_TO
```

### Why no sender-rewrite problem

In the host-side Postfix setup, `mail` sends as `wally@Maria.maria.com` and Yahoo rejects the foreign `MAIL FROM`. That required `sender_canonical_maps` with a regexp catch-all to rewrite every sender.

Alertmanager doesn't have this problem — it sets `smtp_from` explicitly in its config. The `MAIL FROM` envelope sender is `walidahmm@yahoo.com` from the start. Yahoo sees the authenticated account matches the sender and accepts it. No rewriting needed.

### User setup (for someone cloning this repo)

```bash
# 1. Copy the example and fill in your credentials
cp .env.secrets.example .env.secrets
vi .env.secrets

# 2. (Optional) Edit .env to change SMTP_RELAYHOST or SMTP_FROM

# 3. Deploy — alertmanager config is rendered automatically
make up

# 4. To disable alerting, set ENABLE_ALERTING=false in .env
#    make up will fall back to the stub config (no email)
```
